## Story

作为操作者，
我希望 `--all` 模式运行后能生成一份聚合结论文档，包含所有数据源的采集状态、实际字段清单、11 张报表映射模板和 AI 分析提示语，
以便我能将该文档直接交给 AI 完成字段映射分析，替代原有的人工逐源对标工作流。

## Acceptance Criteria

1. **Given** `reporter.py` 的 `_render_raw_report()` 被调用；**When** 报告生成完成；**Then** raw 报告**仅包含**实际返回字段表格（字段名/类型/示例值/可空），**不包含**"需求字段（待人工对照）"区块

2. **Given** `--all` 模式运行完成，所有 source 结果已收集；**When** 调用 `write_aggregate_report(source_results)`；**Then** `reports/all-sources-aggregate.md` 被创建，包含 4 部分：
   - Part 1: 数据源采集状态汇总表（数据源名/状态/错误信息）
   - Part 2: 各数据源实际字段清单汇编（按 source 分组，多表 source 按表分 section）
   - Part 3: 11 张报表字段映射模板（每张报表一个表格，字段为行，含"数据源映射"列）
   - Part 4: AI 分析提示语（指导 AI 使用真实字段完成映射分析）

3. **Given** `--source xxx` 模式运行；**When** 运行完成；**Then** 仅生成该 source 的 raw.md，**不**生成聚合文档

4. **Given** `reporter.py` 中原有 `init_validation_report()` 函数；**When** 重构完成；**Then** 该函数及所有调用点（`sources/awin.py`、`sources/social_media.py`、`sources/youtube_studio.py`、`validate.py`）均已移除

5. **Given** `config/field_requirements.yaml`；**When** 重构完成；**Then** 采用 `reports` 列表结构，包含 11 张报表完整定义（`report_name` + `fields` 列表），每个字段为纯字符串

6. **Given** `validate.py` 的 `_run_source()` 函数；**When** 重构完成；**Then** 返回值为 `Dict`（含 `success`/`status`/`error`/`fields`），`--all` 模式在调度循环结束后调用 `reporter.write_aggregate_report(source_results)`

7. **Given** 所有测试文件适配完成；**When** 运行 `pytest tests/test_reporter.py tests/test_validate.py tests/test_structure.py tests/test_field_requirements.py tests/test_e2e.py`；**Then** 全部通过（当前全量 450+ 个测试，本 story 涉及的 5 个文件约 103+ 个）

### validate.py `_run_source()` 新返回值

```python
def _run_source(source_name: str, module_path: str, table: Optional[str] = None) -> Dict:
    result: Dict = {"success": False, "status": "未知", "error": None, "fields": {}}
    # ... 执行后收集 fields:
    # 非多表: result["fields"][source_name] = fields  （用 source_name 作 key，非 None）
    # 多表:   result["fields"][table_name] = fields

    # write_aggregate_report 是额外的全局聚合产物，不替代单源报告

```

### 测试 mock 策略变更（关键）

**旧策略**（Story 5.1/5.2）：`patch("validate.SOURCES", mock_sources)` — 传入 MagicMock 对象作为 source 模块

**新策略**：`patch("validate.importlib.import_module", side_effect=_make_import_patcher(mock_modules))` — 保持 SOURCES 为字符串路径，拦截 `importlib.import_module` 调用返回 mock

原因：`_run_source()` 改为调用 `importlib.import_module(module_path)` 延迟加载模块，旧策略传入 MagicMock 会导致 `importlib.import_module(MagicMock)` 抛 TypeError。

```python
REAL_SOURCES = {
    "triplewhale": "sources.triplewhale",
    "tiktok": "sources.tiktok",
    # ...10 个 source
}

def _make_import_patcher(mock_modules: dict):
    def _import(module_path):
        if module_path in mock_modules:
            return mock_modules[module_path]
        raise ImportError(f"No mock for {module_path}")
    return _import
```

### triplewhale / tiktok 多表 mock 注意事项

`validate.py` 中 triplewhale 和 tiktok 走多表分支（`module.TABLES`）。默认 mock 必须设置 TABLES 属性：

```python
if name == "triplewhale":
    mock.TABLES = ["default_table"]
    mock.fetch_data_profile.return_value = {
        "table_name": "default_table", "date_column": None, ...
    }
elif name == "tiktok":
    mock.TABLES = ["default_table"]
```

### 聚合文档渲染细节

**Part 2（字段清单汇编）：**
- 非多表 source：用 source_name 作为 section 标题（如 `### dingtalk`）
- 多表 source：用 `### {source_name} — {table_name}` 作为 section 标题
- 采集失败的 source：显示 `（{status}，无字段数据）`，不渲染字段表格

**Part 3（报表映射模板）：**
- 每张报表一个表格，列定义：`| 字段名 | 数据源映射 |`
- "数据源映射"列默认为空（由 AI 在 Part 4 提示语指导下填写）
- 字段来源：`field_requirements.yaml` 的 `reports[].fields[]`

**`write_triplewhale_data_profile()` 保留：** triplewhale 的 raw.md 仍包含数据概况区块，该函数不在本次移除范围内。

### AI 分析提示语要点

聚合文档 Part 4 内嵌的 AI 提示语需指导：
- 使用 4 态映射：可映射 / 需转换 / 缺失 / 待人工确认
- 只使用 Part 2 中列出的真实字段名，禁止编造
- 采集失败的数据源标注"数据暂缺"
- 中文引号嵌套在 Python 字符串中需用单引号包裹（避免 SyntaxError）
