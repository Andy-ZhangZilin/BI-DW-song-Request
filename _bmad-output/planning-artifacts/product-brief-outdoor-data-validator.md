# Product Brief: outdoor-data-validator

## 执行摘要

大户外（Piscifun / TideWe）的业务数据分散在 10 余个平台：TikTok Shop、TripleWhale、Mailchimp、钉钉多维表、YouTube、联盟营销后台……每个平台有独立的 API 协议、字段命名规范和数据口径。当前业务团队无法确认这些数据源能否按需输出指定字段，更无法支撑统一的跨渠道报表体系。

**outdoor-data-validator** 是一个面向运营和业务团队的内部数据源验证工具。它系统性地对接各平台 API，抽取样本数据，对标《数据表需求》中定义的字段清单，输出逐数据源的可行性验证文档——明确回答"哪些字段可以取到、哪些取不到、有哪些差距需要补齐"。

第一期目标：在 2026-04-08 前完成核心数据源（TikTok、TripleWhale、钉钉）的字段验证，支撑 2026-04-24 上线的利润表与销售表现报表。

---

## 问题

### 现状痛点

大户外运营体系正在从分散的手工汇报迈向数据驱动的报表平台。但在这个转型关键节点，团队面临一个基础性挑战：

**没有人确切知道这些数据能不能被取到。**

- TikTok Shop API 要求 OAuth 签名 + Shop Cipher，字段名与业务口径不一致（订单金额 ≠ GMV）
- TripleWhale 部分表（如 `social_media_pages_table`）能否按字段输出尚未验证
- 钉钉多维表的公式字段和复杂关联字段通过 API 读取存在兼容性风险
- CartSee 已确认无公开 API，Mailchimp 已确认接入，但 EDM 指标字段是否覆盖需求尚不清晰
- PartnerBoost 无完整 REST API，依赖第三方工具（Strackr/Affluent），稳定性未知

**如果不在开发前验证，将导致：**
- 报表延期：开发中途发现字段缺失，需推翻返工
- 需求失真：业务按"能取到的字段"折中定义需求，而非按真实需求倒推数据
- 沟通失焦：技术与业务对"字段能否输出"争论不休，缺乏事实依据

### 验证缺口的规模

当前《数据表需求》定义了 5 类核心报表，涉及约 **10 个数据源 × 多份字段清单**，无一已经过系统性 API 层验证。一期报表上线截止日为 2026-04-24，窗口期仅 3 周。

---

## 解决方案

outdoor-data-validator 以"字段可行性验证"为核心，分三步交付：

**Step 1 — API 接入验证**
对每个数据源完成认证配置（OAuth / API Key / Access Token），调通核心端点，确认数据可以被拉取。

**Step 2 — 字段对标**
将 API 返回的原始字段与《数据表需求》中定义的字段逐一比对：
- ✅ 直接可用：字段存在且口径一致
- ⚠️ 需转换：字段存在但需聚合/计算（如 GMV = Σ order_item_price × qty）
- ❌ 缺失：字段无法通过 API 获取，需替代方案

**Step 3 — 验证报告输出**
为每个数据源生成结构化验证文档，明确标注字段覆盖状态、差距说明、建议的解决方案（替代 API / 手动维护 / 跳过）。

**一期不含**：数据入库、ETL 管道、BI 可视化。验证文档本身即是交付物，作为后续开发的输入依据。

---

## 服务对象

**主要用户：运营/业务团队**（需求方）
- 需要确认"我要的数据能不能来"，目前无法自行验证 API
- 关注字段覆盖率和数据可用性，不关心技术实现细节
- 期望：拿到一份清晰的"能用 / 不能用 / 这样用"清单

**次要用户：数据/技术团队**（执行方）
- 需要明确的接入规范和字段映射，作为开发 ETL 管道的前置文档
- 关注 API 认证方式、Rate Limit、数据刷新频率
- 期望：验证过程可复现，结论有技术依据

---

## 差异化

这不是一个通用的数据集成平台，也不是可商业化的 SaaS 产品。它的价值来自**针对性**：

- **字段清单驱动**：验证逻辑完全对标《数据表需求》，不做泛化扫描
- **结论可操作**：每个差距都配有明确的解决建议，而非只报告问题
- **阶段锁定**：与报表上线排期绑定，按优先级分期验证，不做过度设计
- **内部沉淀**：验证文档沉淀为团队知识资产，后续迭代可复用

## 成功标准

| 指标 | 一期目标（2026-04-08） |
|------|----------------------|
| 核心数据源接入率 | TikTok API、TripleWhale、钉钉多维表 100% 完成 API 认证 |
| 字段验证覆盖率 | 利润表 + 销售表现报表所需字段 100% 完成对标 |
| 差距明确率 | 每个"缺失"字段均附有替代方案或明确结论（可用 / 需转换 / 缺失+建议） |
| 文档交付 | 每个数据源一份验证报告，使用统一模板，可直接作为 ETL 开发输入 |
| 凭证到位 | TikTok App Key、TripleWhale shopId、钉钉 AppKey/AppSecret 在 04-04 前到位 |

二期（2026-04-20 前）：营销表现报表所需数据源（Mailchimp、联盟后台、社媒后台）完成验证

三期（2026-06-05 前）：TikTok 达人/联盟数据、产品级营销数据完成验证

## 范围

### 一期范围（IN）

| 数据源 | 验证内容 | 对应报表 |
|--------|---------|---------|
| TikTok Shop API | 订单、商品表现、达人订单、视频表现 | 利润表、TikTok 销售 |
| TripleWhale | Pixel Joined、Pixel Orders、Sessions、Ads、Social Pages | 利润表、营销表现 |
| 钉钉多维表 | 红人支付、KOL 管理总表（TideWe/Piscifun）、大户外一张表 3.0 | 营销表现 |
| Mailchimp | EDM 投送量、打开率 | 营销表现 |

### 明确不在范围（OUT）

- 数据入库 / ETL 管道开发
- BI 报表界面开发
- PartnerBoost / Awin 联盟数据（非一期报表依赖）
- 社媒内容详情抓取（YouTube、Instagram — 非本期核心需求）
- TikTok Business 广告 API（需审批流程，延期风险高）

### 已知风险与降级预案

| 风险 | 概率 | 降级方案 |
|------|------|---------|
| TikTok OAuth 授权流程耗时 > 预期 | 中 | 先用 Research API（无需完整授权）验证可公开字段 |
| 钉钉多维表公式/关联字段 API 读取不完整 | 中 | 标注为"需手动维护"，记录具体字段限制 |
| TripleWhale 账号缺少特定表权限 | 低 | 联系 TripleWhale 技术支持，明确权限范围后更新结论 |
| 凭证未在 04-04 前到位 | 中 | 优先验证已有凭证的数据源，未到位的标注为"待验证" |

## 后续展望

验证阶段完成后，outdoor-data-validator 的文档资产将直接转化为：

1. **数据接入规范**：为 ETL 管道开发提供字段映射和转换逻辑
2. **数据目录（Data Catalog）**：企业级多平台数据字典，持续维护
3. **自动化验证脚本**：将手工验证转化为可定期运行的数据质量检查
4. **跨品牌复用**：验证逻辑和接入代码可直接复用于新品牌接入

长期目标：构建大户外统一数据层，支持跨品牌、跨渠道的运营决策分析。
