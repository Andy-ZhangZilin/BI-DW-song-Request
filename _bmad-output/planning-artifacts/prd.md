# Product Requirements Document - outdoor-data-validator

**Author:** dadong
**Date:** 2026-04-02

## 执行摘要

大户外（Piscifun / TideWe）的业务数据分散在 10 余个平台，业务团队有明确的报表需求，却无法确认这些平台的数据能否按需输出所需字段。在报表开发正式启动之前，这一不确定性是最高优先级的风险。

**outdoor-data-validator** 是一个内部数据源字段可行性验证工具。它系统性地对接各平台数据源，抽取样本数据，逐字段比对《数据表需求》中定义的字段清单，输出结构化验证报告——明确回答"哪些字段可以取到、哪些取不到、差距如何补齐"。

工具覆盖两类数据获取路径：**API 直接调用**（TripleWhale、TikTok Shop、钉钉多维表、YouTube，共 4 个）和**账号登录爬虫**（Awin、CartSee、PartnerBoost、社媒后台，共 4 个）。核心服务对象为运营/业务团队（直接获取字段可用性结论，无需手动到处导数据）和数据/技术团队（获取 ETL 开发前置输入）。

### 产品差异化

- **字段清单驱动**：验证逻辑完全对标《数据表需求》，每个结论附带可操作建议（直接可用 / 需转换 / 缺失+替代方案）
- **双路径覆盖**：同时支持 API 直接调用和账号登录爬虫，验证结论完整覆盖全部 8 个数据源
- **排期绑定**：与报表上线排期绑定，一次性完成全部数据源验证，不做超范围设计
- **前置知识优势**：8 个数据源的接入调研于 2026-04-02 完成，直接进入验证实操，效率提升约 30-40%
- **内部知识沉淀**：验证文档直接转化为 ETL 开发输入，后续可复用为数据目录和自动化质量检查

- **日志与错误处理**：接入失败、爬虫异常均有明确错误输出，便于定位和修复

## 用户旅程

### 旅程一：运营同学拿到验证报告（主路径）

**人物：** 小林，运营分析师，负责跟进利润表和销售表现报表的数据需求

**开场：** 小林手里有一份《数据表需求》，列了利润表需要的 20 个字段。她不知道 TikTok Shop API 能不能返回这些字段，只能等技术同学告诉她——但技术同学也不确定，双方陷入反复拉扯。

**过程：** 技术同学运行 outdoor-data-validator，10 分钟内拿到 TikTok Shop 的字段列表，逐条对标需求清单。报告生成后发给小林。

**关键时刻：** 小林打开 Markdown 报告，看到一张清晰的表格：GMV 标 ⚠️（需聚合计算），退款金额标 ✅，达人佣金标 ❌（替代方案：从财务对账单推算）。她第一次在开发前就清楚地知道哪些字段能用、哪些要妥协。

**结局：** 小林直接用报告调整了需求优先级，技术团队拿到报告开始 ETL 开发，不再有"到底有没有这个字段"的争议。

---

### 旅程二：技术同学接入 API 数据源（执行路径）

**人物：** 大明，数据工程师，负责运行验证脚本

**开场：** 大明需要验证钉钉多维表的字段。他不确定企业内部应用的 Access Token 是否已就绪，也不确定 API 能否读取公式字段。

**过程：** 大明在 `credentials.md` 中找到 AppKey 和 AppSecret，配置到 `.env` 文件，运行钉钉验证脚本。脚本自动获取 Token、读取指定工作簿、输出所有 Sheet 的字段列表。

**关键时刻：** 脚本日志显示"公式字段返回计算值，但关联字段返回空"。大明立刻定位到问题：关联字段需要额外权限申请。报告中该字段标注 ❌（需申请特殊权限，建议手动维护）。

**结局：** 技术团队在 30 分钟内完成钉钉的完整字段验证，验证报告直接进入文档体系。

---

### 旅程三：爬虫数据源验证（异常路径）

**人物：** 大明，同上

**开场：** 大明要验证 Awin 的字段，没有公开 API，只能用 Playwright 登录后台抓取报表数据。

**过程：** 大明运行 Awin 爬虫脚本，配置账号密码，脚本自动登录、导航到报表页面、提取字段结构。

**关键时刻：** 脚本报错——"登录失败，验证码弹出"。大明手动完成验证码后重新运行，脚本继续执行并成功输出字段列表。

**结局：** 验证报告完成，字段结构已记录。大明在 README 中补充"Awin 爬虫偶有验证码，需手动干预"，后续同学接手无障碍。

## 领域特定需求

### 凭证安全

- 所有凭证（API Key、账号密码、Access Token）仅从 `.env` 文件读取，不硬编码在代码中
- `.env` 加入 `.gitignore`，严禁提交到版本控制；项目提供 `.env.example` 模板
- 日志和验证报告输出中不得包含完整凭证信息

### 爬虫合规

- Awin、CartSee、PartnerBoost、社媒后台的自动化登录抓取方案，实施前需查阅各平台 ToS 确认是否允许
- 合规结论记录在各数据源验证报告中，作为后续 ETL 开发依据

### TikTok Shop API 技术约束

- **时间戳参数**：签名参数 `timestamp` 需使用略靠前的时间，不可用当前精确时间戳，参照历史 Java 实现文档的偏移处理逻辑，否则服务端验签失败
- **Access Token**：每次调用 API 前重新获取，不做本地缓存
- 其他签名参数规范以历史 Java 文档为准，Python 实现需严格对照

## 开发工具技术需求

### 技术栈

- **语言**：Python 3.x
- **运行方式**：本地命令行，单机执行，无需服务器部署
- **依赖管理**：`requirements.txt`（`pip install -r requirements.txt` 一键安装）
- **爬虫引擎**：Playwright

### 核心入口

统一入口脚本 `validate.py`，支持按数据源名称或全量运行：

```bash
python validate.py --source triplewhale   # 单个数据源
python validate.py --all                  # 全部数据源
```

每个数据源有独立模块（`sources/triplewhale.py` 等），统一由 `validate.py` 调度。运行时日志实时打印到控制台，异常详细输出。

### 字段需求配置

字段需求清单以 YAML 配置文件管理（`config/field-requirements.yaml`），按数据源和报表分组定义所需字段，无需改代码即可更新。

### 报告输出

- 输出目录：`reports/`
- 命名规范：`reports/{datasource}-validation.md`
- 每次运行覆盖更新同名文件

## 项目范围与阶段规划

### MVP 策略

**MVP 定位**：问题解决型 MVP — 一次性完成全部 8 个数据源的字段可行性验证，输出可直接作为 ETL 开发输入的验证报告。

**资源配置**：1 名执行人，本地 Python 3.x + Playwright 环境，无需额外基础设施。

### MVP 功能集（一期全量交付）

- 全部 8 个数据源接入认证（4 个 API + 4 个爬虫）
- 样本数据抓取，完整输出字段列表（字段名、类型、示例值）
- 对标 `config/field-requirements.yaml`，自动标注字段匹配状态
- 输出 Markdown 验证报告，统一三态标注（✅ / ⚠️ / ❌）
- 日志与错误处理，爬虫验证码手动干预支持
- 统一入口 `validate.py --source {name}` 和 `--all`

### Post-MVP 功能

- 字段需求配置热更新（不改代码）
- 多数据源批量运行汇总统计
- 报告自动 diff（与上次结果对比）

### 远期愿景

- 转化为定期运行的数据质量检查任务
- 跨品牌配置复用（TideWe 等）
- 大户外统一数据目录（Data Catalog）

### 风险与应对

| 风险 | 概率 | 应对方式 |
|------|------|---------|
| TikTok OAuth 首次授权流程耗时 | 中 | 凭证已就绪，参照 Java 文档实现签名，注意时间戳偏移 |
| 爬虫遇验证码 / 登录反制 | 中 | 手动干预机制，不静默失败，记录在 README |
| 钉钉关联字段 API 读取不完整 | 中 | 标注"需手动维护"，不阻塞报告交付 |
| 社媒后台账号未到位 | 中 | 优先跑已有凭证数据源，社媒标注"待跟进" |

## 功能需求

### 1. 工具运行控制

- **FR1**：操作者可通过命令行指定单个数据源名称运行该数据源的验证流程
- **FR2**：操作者可通过命令行一次性运行全部数据源的验证流程
- **FR3**：系统在每次运行时将执行过程打印到控制台，包含数据源名称、当前操作步骤和最终状态

### 2. 凭证与配置管理

- **FR4**：操作者可通过 `.env` 文件配置所有数据源的认证凭证（API Key、账号密码、Token 等）
- **FR5**：系统提供 `.env.example` 模板，列出所有需要填写的凭证字段
- **FR6**：操作者可通过 YAML 配置文件定义各数据源的字段需求清单，无需修改代码

### 3. 数据源接入认证

- **FR7**：系统可完成 API Key 方式的认证接入（TripleWhale、YouTube）
- **FR8**：系统可完成 OAuth 2.0 + HmacSHA256 签名方式的认证接入（TikTok Shop）
- **FR9**：系统可完成钉钉开放平台 Access Token 方式的认证接入（钉钉多维表）
- **FR10**：系统可完成账号密码登录方式的浏览器自动化接入（Awin、CartSee、PartnerBoost）
- **FR11**：系统在接入失败时输出具体错误信息，指明失败原因（认证错误、网络超时等）
- **FR12**：系统在爬虫接入遇到验证码或登录拦截时，中断执行并提示操作者手动干预

### 4. 样本数据抓取

- **FR13**：系统可从每个数据源抓取至少一条真实样本记录
- **FR14**：系统可完整列出样本记录中所有返回的字段（字段名、数据类型、示例值）
- **FR15**：系统可将原始字段列表输出到对应数据源的验证报告文件

### 5. 字段对标分析

- **FR16**：系统可将数据源返回的字段与配置文件中定义的需求字段逐一比对
- **FR17**：系统可对每个需求字段自动标注匹配状态（精确匹配 / 模糊匹配 / 未匹配）
- **FR18**：操作者可在报告中为每个字段填写人工判断结论（✅ 直接可用 / ⚠️ 需转换 / ❌ 缺失）
- **FR19**：操作者可在报告中为"需转换"字段填写转换逻辑说明
- **FR20**：操作者可在报告中为"缺失"字段填写替代方案或明确结论

### 6. 验证报告生成

- **FR21**：系统可为每个数据源生成独立的 Markdown 验证报告，输出到 `reports/` 目录
- **FR22**：验证报告包含：数据源基本信息、接入状态、原始字段列表、字段对标结果表
- **FR23**：字段对标结果表使用统一三态标注（✅ / ⚠️ / ❌），每条结论附带说明
- **FR24**：每次运行覆盖更新同名报告文件，保持报告内容为最新验证结果
- **FR25**：系统可生成符合统一模板格式的报告，可直接作为 ETL 开发的输入文档

### 7. 日志与错误处理

- **FR26**：系统对每个操作步骤输出结构化日志（格式：`[数据源] 操作描述 ... 成功/失败`）
- **FR27**：系统在发生异常时输出完整错误信息，包含错误类型、位置和建议排查方向
- **FR28**：系统在单个数据源失败时不影响其他数据源的验证流程继续执行

### 8. 项目可交接性

- **FR29**：项目提供 README 文档，说明运行环境要求、安装步骤和各数据源的运行命令
- **FR30**：项目通过 `requirements.txt` 管理所有依赖，支持一键安装

## 非功能需求

### 安全性

- 凭证仅从 `.env` 文件读取，不硬编码在代码中
- 日志和报告输出中不包含完整凭证信息（Access Token、密码等）
- `.env` 在 `.gitignore` 中明确排除

### 可靠性

- 相同输入在不同时间运行，字段列表输出结果一致
- 单个数据源失败不中断其他数据源的验证流程
- 爬虫异常不静默失败，有明确错误提示和退出状态码

### 可维护性

- 新增数据源只需在 `sources/` 目录添加新模块，无需修改核心入口逻辑
- 每个数据源模块遵循统一接口规范（接入 → 抓取 → 输出字段列表）
- 关键逻辑（TikTok 签名算法、时间戳偏移）有代码注释说明

### 性能

- 单个数据源的样本抓取和报告生成在正常网络条件下 60 秒内完成
- 全部数据源批量运行总时长不作硬性要求，以实际网络和平台响应为准
